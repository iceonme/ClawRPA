- [微博风控] 机械快速刷新/重复发送同一 chat URL 会触发微博 HTTP 418，页面表现为“该网页无法正常运作 / HTTP ERROR 418”，约 3 分钟恢复。当前微博 chat 策略为场景级降频：每目标至少 20 秒，打开后等 5-7 秒，慢速输入，发送前等 1-5 秒，发送后等 5-10 秒。
- [降频验证] 测试地址 `https://api.weibo.com/chat/#/chat?to_uid=1830582752`。同一进程、共享 `WeiboChatPacer` 连续发送 10 条测试消息成功，未触发 418。
- [编码经验] PowerShell here-string 或命令行直接传中文给 Python 时可能变成 `????`，导致真实消息乱码。后续中文消息必须从 UTF-8 文件读取，或在 Python 代码内用 Unicode escape 构造。已用测试地址发送“编码测试：杭州蔡依林...”验证 UTF-8/Unicode 路线可正常发中文。
- [Lead 规则] 旧 lead 规则过窄，只认 `求票/收票/蹲票/没抢到/回流` 等明确票务词。已放宽到两档：明确票务词是强信号；`想去/好想去/想看/去现场/抽我/看看我/带我/祝我抢到/杭州见/西湖见` 等作为强意愿待确认，低置信度、需复核。
- [Lead 噪音] 放宽后召回提升，但新闻/媒体帖评论中会出现大量“批评恶意抢票/讨论退票规则”的噪音。筛选时要排除媒体号、纯规则讨论、批评别人抢票的评论；优先保留第一人称需求，如“我没抢到”“我想去”“坐等回流”“想要去的我抢不到”。
- [杭州蔡依林实测] `杭州 蔡依林` 600 秒内旧规则拿到 5 条；放宽规则后约 5 分 40 秒拿到 15 条。筛选后建议优先触达 P1/P2，P3 粉丝到场意愿只作为补量池。
- [指定帖补量] 用户指定帖子 `https://weibo.com/1742727537/QBfL0w4FW`。深扫评论可找到“杭州见/杭州想听/杭州场”等强意愿评论，但很多并非缺票，需要谨慎作为 P3。
- [话术边界] 首次触达消息适合脚本拼接，不必调用 agent，省 token 且更稳定。后续对话再交给 OpenClaw/agent。首次模板来自 `C:\Users\iceon\.openclaw\workspace-piaoduoduo\task_share\话术.md`，字段主要填 `event_query` 和 `signal_text/context`。
