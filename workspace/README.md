# ClawRPA 项目工作标准 & workspace 总览

> 版本：v1.2 | 生效：2026-05-20 | 维护：TIX1号
> 本项目所有文件操作、任务跟踪、决策记录均须遵循本标准。

---

## 一、目录结构总览

```
clawRPA/
├── code/                         # 代码仓库
│   ├── browser/                  #   网页 RPA 脚本
│   ├── mobile/                   #   手机端脚本（桥接 + AutoX.js）
│   └── tix_workflow/             #   票务工作流脚本
├── workspace/                    # 工作空间（所有非代码产出）
│   ├── 01_data/                  #   原始数据（D0XX）
│   ├── 02_knowledge/             #   知识库/研究笔记（K0XX）
│   ├── 03_insight/               #   技术方案决策（I0XX）
│   ├── 04_tasks/                 #   任务看板（T0XX）
│   └── memory/                   #   工作日志/交接
├── test_env/                     # Python venv
├── skills/                       # 独立技能脚本
├── .agents/                      # 工作流定义
└── readme.md                     # 项目总说明
```

**原则：** `code/` 放可执行代码，`workspace/` 放知识资产。不混放。

## 二、工作空间各目录规范

### `01_data/` — 原始数据

存放爬取、探测、API 获取的**原始数据**，不做分析、不做汇总。

| 要求 | 说明 |
|------|------|
| **命名** | `D0XX_描述性名称.ext`（D = Data） |
| **内容** | 原样保存，只清洗格式不篡改语义 |
| **Git** | 小文件可提交，大文件标注来源路径 |
| **示例** | `D001_openclaw浏览器skill-公众号.md` |
| **子目录** | 可按来源建子目录（如 `debug/`、`redroid/`） |

> 不要在大文件中夹带分析结论。结论放在 02_knowledge 或 03_insight。

---

### `02_knowledge/` — 知识库

存放**研究笔记、调研报告、技术分析**，可供多人参考。

| 要求 | 说明 |
|------|------|
| **命名** | `K0XX_主题_YYYYMMDD.md`（K = Knowledge） |
| **前缀** | 顺序编号，跨主题不重复 |
| **内容** | 背景、分析、结论、参考链接 |
| **示例** | `K005_RPA_Techniques_Comparison_20260415.md` |

---

### `03_insight/` — 技术决策记录

存放**关键方案对比、技术选型推理、架构决策**。目的是给未来的自己和队友看"当时为什么这么选"。

| 要求 | 说明 |
|------|------|
| **命名** | `I0XX_主题_YYYYMMDD.md`（I = Insight） |
| **前缀** | 顺序编号，与任务编号无关 |
| **标准结构** | 见下方模板 |

```markdown
# I0XX — 主题

> 日期：YYYY-MM-DD | 作者：TIX1号
> 前情：[链接到前序 Insight 或 Task]
> 决策：[一句话结论]

## 问题描述

[背景、痛点、要解决什么]

## 候选方案

| 方案 | 优点 | 缺点 |
|------|------|------|
| 方案A | ... | ... |
| 方案B | ... | ... |

## 选择理由

[为什么选这个]

## 弃用路径

[被否决的方案及其原因，供以后参考]

## 关联 Task

- T0XX: [什么任务]
```

---

### `04_tasks/` — 任务看板

项目**任务管理中心**。所有任务在此跟踪。

详见 `04_tasks/README.md`（结构设计 + 工作要求）和 `04_tasks/board.md`（状态总看板）。

---

### `memory/` — 工作日志

| 文件 | 用途 |
|------|------|
| `YYYY-MM-DD.md` | 每天的工作记录 |
| `handover.md` | 关键交接信息 |
| `board.md` | 遗留的历史看板（票牛项目） |

> workspace/memory/ 记录"每天发生了什么"。
> 任务过程的详细踩坑记在对应 T0XX/memory.md 里。

## 三、全局命名规范

| 目录 | 编号 | 日期格式 | 示例 |
|------|------|----------|------|
| `01_data/` | `D0XX` | 无强制要求 | `D001_description.md` |
| `02_knowledge/` | `K0XX` | `YYYYMMDD` | `K005_RPA_Techniques_Comparison_20260415.md` |
| `03_insight/` | `I0XX` | `YYYYMMDD` | `I004_Phone_Automation_Approaches_20260519.md` |
| `04_tasks/` | `T0XX` | 目录名无日期 | `T002_Phone_Bridge_RPA/` |

> 编号不跨目录关联。I003 和 T003 之间无数字映射关系，关联通过文件内链接实现。

## 四、文件联动规则

### 行为铁律：写文档是完成的一部分 🎯

> 每完成一个子任务或里程碑，必须**立即**更新相关文档。
> **文档滞后 = 任务未完成。** 不能留到"以后补"。

这是整个标准体系里最重要的行为规则。技术实现是一回事，文档同步是完成它的必要条件。

### 联动对照表

| 关系 | 规则 |
|------|------|
| **写文档是完成的一部分** | 完成子任务/里程碑 → 立即更新 board.md（状态）+ memory.md（记录）+ 关联同步。文档没更新 = 任务没做完。 |
| **Insight ↔ Task（双向关联）** | 一个 Insight 驱动多个 Task，一个 Task 引用多个 Insight/其他 Task。如果 A 引用了 B，B 的关联里也要引用 A。必须双向同步。 |
| **关联双向同步** | 每个文档底部的 `## 关联` 区必须实时同步。新加了一条引用，就去被引用的文档里加上回链。 |
| **看板 ↔ board.md** | **最核心：** 04_tasks/board.md（总看板）必须和 T0XX/board.md（子任务状态）同步。新增目录 → 立刻更新总看板。 |
| **plan.md 不跟踪状态** | plan.md 只写方案说明，不写进度。进度和状态唯 board.md 唯一来源。 |
| **memory.md 不跟踪状态** | memory.md 是复盘日记，不取代 board.md 的状态跟踪。 |
| **Task → 代码** | Task 产出代码，在 board.md 子任务表或 files/ 中注明产出路径。 |

### 状态同步检查清单

每次改动任务后，逐条确认：

- [ ] T0XX/board.md 的子任务状态已更新
- [ ] 04_tasks/board.md 的总看板已同步
- [ ] 相关文档的 `## 关联` 区已双向同步
- [ ] 如果新增 T0XX 目录，AGENTS.md 项目上下文已更新
