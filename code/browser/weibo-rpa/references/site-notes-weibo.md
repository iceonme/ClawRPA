# 微博平台说明

当前任务聚焦：**为后续私信 / 评论回复智能体准备潜客表**。

## 当前重点采集对象
优先采集：
- 评论区里明确求票 / 收票 / 想看但没抢到的人
- 楼中楼继续追问票务信息的人
- 帖子正文直接发求票需求的人

## 关键提取字段
### 用户侧
- 昵称
- uid / 唯一标识
- 主页链接

### 帖子侧
- 帖子 id
- 帖子链接
- 帖子作者
- 帖子发布时间
- 帖子摘要

### 信号侧
- 评论 / 楼中楼 / 帖子正文文本
- 信号发生位置链接
- 信号发布时间

## 搜索结果页 DOM 观察（已验证）
### 卡片层级
- 搜索结果中每条微博通常对应一个 `div[action-type="feed_list_item"]`
- 卡片容器常见类名为 `card-wrap`
- 卡片上可取到 `mid`

### 操作区
- 卡片内部存在 `card-act`
- 其中操作按钮通常是 `li`
- 评论/回复按钮点击后，会动态加载评论区域，而不是先跳转新页面

### 评论展开层
- 点击评论/回复后，评论区域会在当前卡片下方动态出现
- 对应区域可观察到 `feed_list_repeat`
- 在这个区域内，`card-more-a` 下面可拿到对应帖子链接

### 详情页入口
- 搜索结果卡片中的时间链接（`.from a`）本身就是一种详情页入口
- 另外，在评论展开后的 `feed_list_repeat` 区域内，也可能出现可用的帖子链接
- 链接常见形式为：
  - `//weibo.com/<uid>/<post-code>?refer_flag=...`
  - 需要在采集时规范化为 `https://weibo.com/...`

## 实现建议
- 不要只靠全文正则扫绝对链接
- 优先按 DOM 定位：
  - `feed_list_item`
  - `card-act`
  - `feed_list_comment`
  - `feed_list_repeat`
  - `card-more-a`
  - `.from a`
- 优先先抽卡片，再点评论，再在展开层中找帖子链接和评论信号

## 关键难点
- 搜索词可能需要多轮扩展，不止一个关键词
- 同一个用户可能在多个帖子下重复出现，需要去重
- 有些高意向表达非常口语化，需要 agent 结合上下文判断
- 有些用户只能拿到昵称，拿不到稳定 uid，需要单独标记风险
- 页面 class 名可能会变，但 `action-type` 一类属性往往更稳

## 常见异常与风控表现
- 搜索结果加载不完整
- 评论区需要展开或滚动后才出现
- 楼中楼需要二次点击展开
- 页面偶发弹窗、登录提示、刷新后重排
