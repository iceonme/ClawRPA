# 架构说明

## 当前架构结论
当前工程已经从最初的 `lead-discovery` 目录骨架，收口为一个更明确的平台型工程：

> **`weibo-rpa`（平台优先）**

在这个工程下，当前第一阶段正式完成的是：

> **`lead` 子能力**

也就是：围绕某个演出/关键词，在微博中发现潜在购票用户，并输出结构化潜客结果。

---

## 为什么当前不采用 `lead-skill / chat-skill` 的跨平台抽象
因为当前真实跑通并验证稳定性的，是：
- 微博工作浏览器
- 微博搜索页
- 微博帖子/正文页入口
- 微博 lead 抓取

而不是微博/小红书两个平台都已经完成了同等深度的实现。

所以现在若过早做：
- `lead-skill`
- `chat-skill`

会带来典型的“过早抽象”问题：
- 业务概念看似统一
- 平台差异被硬塞到大量 `if platform == ...` 逻辑里
- 代码和文档都会迅速膨胀

因此当前架构原则是：

> **先按平台拆，再在平台内部按业务子域拆。**

也就是：
- 顶层：`weibo-rpa`
- 子域：`lead`
- 下一阶段再补：`chat`

---

## 当前 skill 边界
### 负责
- 微博搜索
- 微博搜索结果页正文信号抽取
- 结构化潜客输出
- 运行摘要与产物落盘

### 暂不负责
- 自动私信发送
- 热度分析
- 多平台统一调度
- 验证码/登录绕过
- 评论层完整结构化抽取

---

## 当前目录分层
### `scripts/`
按三层管理：

#### 正式入口
- `run_lead_task.py`
- `run_weibo_probe.py`
- `weibo_collect_test_leads.py`

#### probe
- `scripts/probe/*`

#### archive
- `scripts/archive/*`

### `src/`
保留现有结构，后续逐步向“平台工程 + 子域 workflow”演进：

#### `src/core/`
跨任务复用的基础能力：
- 浏览器连接
- CDP attach
- I/O 协议
- waits / artifacts / config

#### `src/flows/`
当前版本的业务流程层：
- 微博搜索
- 微博评论规划
- 线索判断
- 线索落盘

#### `src/schemas/`
输入输出结构定义。

#### `src/probes/`
探针实现。

## 当前 OpenClaw 协作模式
仍然保持此前已经对齐的模式：

- `exec` 启动脚本
- 长任务可后台运行
- `process` 查询日志 / 状态
- 大内容全部落到 `runs/`
- stdout 只输出少量结构化状态/结果

也就是：

> **脚本做真实工作，文件保存真实产物，stdout 只输出少量结构化状态/结果索引。**

这仍然是当前最合理的工程模式。

---

## 下一阶段工程化方向
当前版本收口之后，建议按这个顺序继续：

### 1. 状态层
新增 `state store`（优先 SQLite）：
- lead 去重
- 增量刷新
- 首次发现 / 最近发现
- 候选池沉淀

### 2. lead 增量刷新
支持定时任务：
- 不重复导入同一用户
- 新命中更新 `last_seen_at`
- 新线索可入待触达池

### 3. chat 子域
在同一个 `weibo-rpa` 工程下补：
- `chat/send`
- `chat/history`
- 防重复私聊
- task 流转

### 4. 再考虑更高层抽象
只有在微博与其他平台都成熟之后，才考虑是否需要：
- 跨平台 lead skill
- 跨平台 chat skill
- campaign orchestrator

而不是现在就提前统一。
